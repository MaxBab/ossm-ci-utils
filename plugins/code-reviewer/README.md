# code-reviewer Plugin

Multi-phase, subagent-driven code review with auto-maintained project conventions. Works with both **Claude Code** (as a plugin) and **Cursor** (via install script).

## Installation

### Claude Code (via marketplace)

```
/plugin install code-reviewer@ci-utils
```

### Claude Code (standalone)

For use outside the ci-utils marketplace:

```bash
claude --plugin-dir /path/to/ci-utils/plugins/code-reviewer
```

Or add to `~/.claude/settings.json`:

```json
{
  "enabledPlugins": {
    "code-reviewer@local:/path/to/ci-utils/plugins/code-reviewer": true
  }
}
```

### Cursor

Cursor has no plugin system, so an install script transforms the Claude source files and copies them into your project's `.cursor/` directory:

```bash
/path/to/ci-utils/plugins/code-reviewer/install-cursor.sh /path/to/your-project
```

Re-run the script any time the plugin is updated to pick up changes.

## Commands

### `/code-reviewer:setup`

Onboard a project for code review. Analyzes your codebase, discovers existing standards and patterns, and interactively generates reference docs (style guide, testing practices, etc.).

Run this once before your first review:

```
/code-reviewer:setup
```

---

### `/code-reviewer:review`

Run the full multi-phase code review pipeline on your current branch changes:

```
/code-reviewer:review                    # Full pipeline — all three phases
/code-reviewer:review:adversarial        # Adversarial phase only
/code-reviewer:review:style              # Style phase only
/code-reviewer:review:testing            # Testing phase only
```

**Review phases:**

| Phase | Focus | Finding Prefixes |
|-------|-------|-----------------|
| Adversarial | Architecture, correctness, security, API compatibility, edge cases | `BUG-N`, `SEC-N` |
| Style | Convention enforcement against project's style guide | `STY-N` |
| Testing | Test coverage, quality, missing edge cases | `TST-N` |

**How it works:**

1. **Triage** — captures the diff, loads project config and reference docs, groups changes into review units
2. **Parallel dispatch** — sends structured briefs to three specialized review subagents
3. **Consolidation** — merges reports, deduplicates findings, surfaces cross-unit issues, produces verdict
4. **Doc update** — after you respond to findings, considers whether reference docs need updating (always asks first)

---

### `/code-reviewer:ci-review`

Fully autonomous code review for CI pipelines. Runs all three review phases without any user interaction and posts results directly to the PR.

**What it does differently from `/code-reviewer:review`:**

- **No setup required** — If reference docs don't exist, auto-generates them on the fly using the headless-setup skill (no interactive confirmation)
- **No pauses** — Every step runs automatically, no user prompts
- **Auto-posts to GitHub** — Creates a submitted PR review with inline comments + a summary PR comment
- **Convention suggestions** — Instead of interactively updating docs, appends suggested convention updates to the summary comment

**Output posted to the PR:**

1. Inline review comments on specific lines for each finding
2. A summary comment with the full consolidated report, verdict, and convention update suggestions

See [CI / GitHub Actions](#ci--github-actions) below for setup instructions.

## Skills

| Skill | Purpose |
|-------|---------|
| `triage` | Diff analysis and review brief generation |
| `consolidation` | Merge and deduplicate findings from all phases |
| `doc-update` | Living documentation maintenance after reviews |
| `headless-setup` | Non-interactive project onboarding for CI environments |
| `adversarial-review` | Adversarial review phase instructions |
| `style-review` | Style review phase instructions |
| `testing-review` | Testing review phase instructions |

## Per-Project Files

Generated by `/code-reviewer:setup`, stored in your project. Committing these files is recommended — it means setup only runs once and all team members share the same conventions.

```
.claude/
  code-reviewer/
    config.md                             # Project config
    reference/                           # Generated conventions
```

## Configuration

Copy the example config to get started:

```bash
cp <ci-utils>/plugins/code-reviewer/examples/config.md .claude/code-reviewer/config.md
```

Or run `/code-reviewer:setup` to generate it interactively.

### Committed vs. Ephemeral Reference Docs

**Recommended: commit the generated files.** This means `/code-reviewer:setup` only needs to be run once, and all team members share the same enforced conventions without any local setup.

```bash
git add .claude/code-reviewer/
git commit -m "Add code-reviewer reference docs"
```

If you prefer each developer to maintain their own local copy, add `.claude/code-reviewer/` to `.gitignore` instead. The `/code-reviewer:setup` command will ask which you prefer.

## CI / GitHub Actions

Use `/code-reviewer:ci-review` to run fully autonomous code reviews on pull requests. No manual setup step is needed — the command auto-generates reference docs if they don't exist.

### Prerequisites

- **Claude Code CLI** installed in the CI runner
- **`ANTHROPIC_API_KEY`** environment variable set
- **`gh` CLI** authenticated with permissions to read the repo and post PR reviews/comments
- The CI job must **check out the PR branch** (not just the base branch)

### Example GitHub Actions Workflow

```yaml
name: Code Review
on:
  pull_request:
    types: [opened, synchronize]

permissions:
  contents: read
  pull-requests: write

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Run code review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          claude -p "/code-reviewer:ci-review" \
            --plugin-dir ./plugins/code-reviewer
```

If ci-utils is not checked out in the repo, clone it first:

```yaml
      - name: Clone ci-utils
        run: git clone --depth 1 https://github.com/openshift-service-mesh/ci-utils.git /tmp/ci-utils

      - name: Run code review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          claude -p "/code-reviewer:ci-review" \
            --plugin-dir /tmp/ci-utils/plugins/code-reviewer
```

### What Gets Posted to the PR

1. **Inline PR review** (`event: "COMMENT"`) — per-line comments for each finding with structured IDs, severity, description, and fix suggestions. Does not block merge.
2. **Summary PR comment** — full consolidated report with verdict, all findings by severity, cross-unit issues, strengths, open questions, and suggested convention updates.

### Reference Docs in CI

If `.claude/code-reviewer/` is committed to the repo (see [Committed vs. Ephemeral Reference Docs](#committed-vs-ephemeral-reference-docs)), `/code-reviewer:ci-review` uses them directly and skips the discovery step — making CI runs faster and more consistent.

If the files are not committed, `/code-reviewer:ci-review` auto-generates them on each run using the headless-setup skill. This requires no setup but adds discovery time to every CI run.
